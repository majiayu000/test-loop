# Closed-Loop Drill Report — 2026-06-05 v0.2.1

## Purpose

Verify the v0.2.1 parameterised scripts (commit `1ea8651`) work
end-to-end on a non-Swift language, specifically Python's pytest
output. This drill uses a hand-crafted pytest log fed to
`bin/render_report.sh --language python` and
`bin/classify_failures.py --language python` in --collect mode.

It is a *follow-up* to the [v0.2 drill](drill-2026-06-05.md) which
exercised the Swift-shaped scripts. Together they prove the loop
works on at least two languages.

## Setup

1. `bin/check_drift.sh` is parameterised to accept
   `--source-glob`, `--knowledge-base`, and `--language {swift,
   python, go, rust, auto}`. The default remains the caff Swift
   layout.
2. `bin/classify_failures.py` is parameterised to accept
   `--language {swift, python, go, rust}`. It now has 14
   self-tests (10 swift + 4 cross-language), all passing.
3. `bin/render_report.sh` is parameterised to accept
   `--language {swift, python, go, rust}` and `--test-command`.
4. Synthetic pytest log at
   `docs/reports/2026-06-05/log.txt`:
   ```
   ============================= test session starts ==============================
   collected 4 items

   test_example.py F.FF                                                      [100%]

   =========================== short test summary info ============================
   FAILED test_example.py::test_rejects_invalid_input - AssertionError
   FAILED test_example.py::test_happy_path_asserts - AssertionError
   FAILED test_example.py::test_rejects_invalid_input - AssertionError
   FAILED test_example.py::test_with_invalid_email - AssertionError
   ========================= 1 passed, 3 failed in 0.05s ==========================
   ```

## What was run

| Step | Command | Outcome |
| --- | --- | --- |
| 1 | `python3 bin/classify_failures.py --self-test` | `Ran 14 tests in 0.000s / OK` |
| 2 | `python3 bin/classify_failures.py --language python --in log.txt` | JSON with 3 deduplicated failures; `test_rejects_invalid_input` and `test_with_invalid_email` go to `EXPECTED_FAILURE`, `test_happy_path_asserts` goes to `ASSERTION_FAILURE` |
| 3 | `bash bin/render_report.sh --language python --collect` | Wrote `summary.json` (total=4, passed=0, failed=4) and `report.md` listing the four FAILED lines as failing tests |

## Step 2 output (the interesting one)

```json
{
  "failures": [
    "test_rejects_invalid_input",
    "test_happy_path_asserts",
    "test_with_invalid_email"
  ],
  "failures_by_class": {
    "EXPECTED_FAILURE": 2,
    "ASSERTION_FAILURE": 1
  },
  "failures_grouped": {
    "EXPECTED_FAILURE": [
      "test_rejects_invalid_input",
      "test_with_invalid_email"
    ],
    "ASSERTION_FAILURE": [
      "test_happy_path_asserts"
    ]
  }
}
```

Three things to note:

1. **Cross-language deduplication works.** The log emits
   `test_rejects_invalid_input` twice (the per-test line and the
   short summary line). The classifier counts it once.
2. **Cross-language classification works.** `test_rejects_*` and
   `test_*invalid_*` match the Python `EXPECTED_FAILURE` pattern
   (which uses case-insensitive substring matching on `rejects`,
   `invalid`, etc.). `test_happy_path_asserts` has no matching
   keyword and falls through to `ASSERTION_FAILURE`, as designed.
3. **First-seen order is preserved.** `test_rejects_invalid_input`
   comes before `test_happy_path_asserts` because that is the order
   in the log.

## Defects found in this drill

The drill exposed three defects in the v0.2-era scripts that the
parameterisation did not introduce but also did not fix. They are
on the v0.2.2 list.

1. **`summary.json.run_line` falls back to "(no summary line
   found)" on pytest logs.** The regex for the pytest run-summary
   line is `[0-9]+ (passed|failed) in `. The log's actual
   summary is `1 passed, 3 failed in 0.05s`. The regex matches the
   first alternative (`passed`) but then expects `in ` immediately
   after, with no allowance for the comma-separated failure count
   that pytest emits in the same line. Fix: relax the regex to
   `[0-9]+ (passed|failed)(, [0-9]+ failed)? in `.
2. **`summary.json.failures_by_class` is empty even when the
   classifier ran.** `render_report.sh` writes a JSON object that
   contains `failures_by_class: {}` regardless of what the
   classifier produced. The render script does not call the
   classifier (in v0.2 it called `classify_failures.py` but
   discarded the output). Fix: re-instate the merge that was
   present in the caff copy of `render_report.sh`.
3. **`report.md` shows `Result: ✅ PASS` when `failed: 4`.** The
   render uses `$SWIFT_EXIT` (now `$TEST_EXIT`) which is `0` in
   `--collect` mode by design. The `Result:` line should be
   derived from `failed > 0` in addition to the exit code. Fix:
   change the conditional to `[[ $FAILED -gt 0 || $TEST_EXIT -ne 0 ]]`.

These three defects are tracked together and will be fixed in a
single v0.2.2 commit.

## Reproducing this drill

```bash
git clone https://github.com/majiayu000/test-loop
cd test-loop

# 1) self-test
python3 bin/classify_failures.py --self-test

# 2) synthetic drill
mkdir -p docs/reports/2026-06-05
cat > docs/reports/2026-06-05/log.txt <<'EOF'
============================= test session starts ==============================
collected 4 items

test_example.py F.FF                                                      [100%]

=========================== short test summary info ============================
FAILED test_example.py::test_rejects_invalid_input - AssertionError
FAILED test_example.py::test_happy_path_asserts - AssertionError
FAILED test_example.py::test_rejects_invalid_input - AssertionError
FAILED test_example.py::test_with_invalid_email - AssertionError
========================= 1 passed, 3 failed in 0.05s ==========================
EOF

python3 bin/classify_failures.py --language python --in docs/reports/2026-06-05/log.txt
bash bin/render_report.sh --language python --collect

cat docs/reports/2026-06-05/summary.json
```

If any of those steps produces a different outcome, the loop has
regressed since `1ea8651` (this commit's v0.2.1 parameterisation).
